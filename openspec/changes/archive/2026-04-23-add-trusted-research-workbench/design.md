# 可信投研工作台产品模式设计

## 1. 设计目标

本次设计不是继续打磨一个“比赛演示页”，而是把现有 `live console` 明确定义成一套稳定的 `资讯分析工作台产品模式`。

目标有四个：

1. 让用户在首屏快速读懂“发生了什么、为什么重要、依据在哪”
2. 让关键判断、证据摘录、来源上下文形成连续可核验的闭环
3. 让分析前、分析中、分析完成、分析失败与重新分析都保持稳定的工作台语义
4. 让科技感服务于专业感，而不是把页面重新做成展示型 demo 或汇报大屏

非目标：

- 不构建多页面资讯平台
- 不把关系图升级为主入口或通用知识图谱平台
- 不引入历史管理、watchlist 或团队协作系统

## 2. 核心产品定义

页面统一定义为：

**用户导入一条资讯后，在同一工作台内完成事件理解、关键判断、证据核验、主体扫描与分析追溯的单页分析终端。**

这一定义意味着：

- 输入区不再是主角，它是材料收纳与切换器
- 主区必须先服务“读结论”，再服务“看依据”
- 侧栏只增强理解，不承担主结论表达
- 所有技术证据都需要产品化收纳，不能散落成开发者视角字段墙

## 3. 信息架构

### 3.1 桌面端骨架

页面采用宽屏工作台结构：

1. `顶部状态总控条`
2. `左侧收纳式导入仓`
3. `中间主结论区`
4. `右侧态势侧栏`

对应职责如下：

- 顶部状态总控条：展示当前分析对象、来源状态、运行状态、凭证状态与下一步动作
- 左侧导入仓：承载新建分析、粘贴正文、导入链接、更换材料、查看当前材料摘要
- 中间主结论区：固定承载 `事件总览 -> 关键判断 -> 证据摘录`
- 右侧态势侧栏：承载 `辅助关系图 + 核心主体 + 分析元信息入口`

### 3.2 主阅读链路

主区必须按以下顺序组织：

1. `事件总览`
2. `关键判断`
3. `证据摘录`

这是强约束，而不是推荐顺序。

其中：

- `事件总览` 是唯一 Hero 级内容，用一句业务化标题和一段 briefing 建立整体判断
- `关键判断` 只保留 2-3 条，每条都必须回答“为什么重要”
- `证据摘录` 紧跟判断之后，形成“看结论 -> 看依据”的连续链路

## 4. 主区内容模型

### 4.1 事件总览

事件总览只承载首屏最需要的信息：

- 业务化事件标题
- `1-2` 句 briefing
- 固定指标：事件类型、重要性等级、来源方式、凭证状态

以下内容不得直接塞进首屏总览卡：

- 完整摘要
- 大段原文
- 长日志
- 过多技术元信息

### 4.2 关键判断

关键判断必须是“可追溯判断卡”，而不是普通摘要卡。

每张卡应至少包含：

- 判断标题
- 为什么重要
- 关联主体
- 证据锚点或证据入口
- 判断强度或来源提示

没有证据锚点的判断只能作为草稿提示，不能按正式结果展示。

### 4.3 证据摘录

证据摘录是主区固定模块，不可默认折叠到详情层或侧栏。

证据卡应至少包含：

- 对应哪条判断
- 原文摘录
- 来源站点
- 导入状态
- 最小必要上下文或查看上下文入口

详情层可以承载更完整内容，但不能代替首屏证据预览。

## 5. 右侧态势侧栏

右侧区域不再承担“五类信息混装”的杂物柜角色，而是明确收敛为：

- `辅助关系图`
- `核心主体`
- `分析元信息入口`

规则如下：

- 关系图只用于态势浏览，不得与主区竞争首屏焦点
- 核心主体以轻量清单或标签方式呈现，帮助快速扫主体
- 来源、流程、支付、receipt、payloadHash 等信息统一收纳为分析元信息入口

当图数据不足时：

- 降级为主体清单或关系清单
- 不得为了“有图可看”而强行构造复杂图谱

## 6. 状态模型

### 6.1 页面状态

页面按用户视角定义五个主状态：

1. `待导入`
2. `待分析`
3. `分析中`
4. `分析完成`
5. `分析失败`

### 6.2 分析阶段

分析阶段是页面状态下的次级运行信息，例如导入、摘要、主体、关系、图谱整理等。

约束：

- 页面状态与分析阶段必须分开表达
- 页面状态决定首屏骨架、动作按钮与提示文案
- 分析阶段只负责说明“系统正在进行到哪一步”

### 6.3 重新分析与失败恢复

这是本次产品模式化的关键约束：

- 首次进入页面且没有运行中任务时，工作台可以默认空白
- 一旦用户基于已有结果发起重新分析，旧结果必须保留到新结果成功替换为止
- 如果新一轮分析失败，页面应继续保留旧结果并叠加失败反馈，而不是整页清空
- 失败后用户必须在当前工作台低成本执行 `重新分析` 或 `重新导入`

## 7. 视觉语言

视觉基调统一为：

**冷静克制的专业未来感**

允许使用的科技感表达：

- 高精度网格与分栏
- 低饱和冷色体系中的单一强调色
- 微弱结构线、状态线、边框与标签系统
- 等宽数字、时间戳对齐、轻量状态动效
- 焦点区域的局部弱发光或弱渐变

必须限制的表达：

- 大面积霓虹光效、扫描线、流动光带、粒子特效
- 高饱和黑底亮色组合
- 过重玻璃拟态和大面积毛玻璃叠层
- 抢主视觉的关系图与仪表盘式组件
- 展示型 hero、宣传式文案和看板 KPI 化表达

## 8. 产品约束与验收

### 8.1 硬约束

1. 主阅读链路不可被打断
2. 所有关键判断都必须能追到证据与来源上下文
3. 关系图只能辅助理解，不承担主阅读任务
4. 重新分析与失败恢复不得清空旧结果
5. 视觉、文案、动效都必须服务可信、克制、可连续阅读

### 8.2 验收重点

- 首屏无需滚动即可看懂当前对象、当前状态、主结论入口和下一步动作
- 分析完成时，首屏先看到事件总览与关键判断，而不是关系图或装饰视觉
- 每条关键判断都能对应到证据摘录
- 失败或重跑时，用户仍留在同一工作台上下文里
- 页面整体更像可信分析终端，而不是资讯看板或 demo 页面

## 9. 第二轮强化重构决策

### 9.1 页面定位

在第一轮产品模式化基础上，`/demo/live` 本轮进一步拍板为：

**结果优先的情报分析台**

这意味着页面不再保留“工作台说明页 + 输入页 + 结果页”的过渡痕迹，而是直接呈现一个可以长期使用的分析终端。

### 9.2 视觉方向

本轮不走暗色赛博风，而采用：

**浅底矿物质科技感**

具体要求：

- 暖灰白底、石墨正文、深青点缀、少量黄铜强调
- 科技感来自精密网格、边界线、结构秩序与轻量状态反馈
- 禁止大面积玻璃拟态、舞台化光斑、霓虹和“看板大屏”式炫技组件

### 9.3 布局重拍板

桌面端首屏固定为四段式：

1. `顶部工具带`
2. `宽主阅读区`
3. `稳定右侧情报侧栏`
4. `右侧详情抽屉`

与第一轮不同的是：

- 输入入口默认退入工具带中的触发器或可展开导入仓，不再作为左侧一级内容区常驻首屏
- 预置资讯只作为启动器，不再以卡片墙形式占据主内容层
- 主区宽度必须明显大于侧栏，确保 `事件总览 -> 关键判断 -> 证据摘录` 成为唯一阅读中轴

### 9.4 首屏结构

首屏只允许出现以下主内容：

- 顶部工具带：当前会话状态、来源类型、导入状态、总成本、成功调用数、主操作
- 主区事件总览：headline、1-2 句 briefing、事件类型、重要性、来源方式、总成本
- 主区关键判断：最多 3 张判断卡
- 主区证据摘录：最多 2 组首屏证据预览

以下内容必须退后：

- 完整摘要
- 预置样本大列表
- 长说明文案
- 流程、payload、receipt 长列表

### 9.5 侧栏与详情层

右侧情报侧栏固定顺序为：

1. `导入来源`
2. `核心主体`
3. `辅助关系图`
4. `分析凭证`
5. `分析流程`

规则：

- 侧栏 sticky，桌面端宽度不超过总宽度的 `35%`
- 关系图只能辅助，不得进入主区中轴
- 详情层统一采用桌面右侧抽屉、移动端全屏面板
- 所有长文本、完整证据上下文与 payload/receipt 细节都进入详情层或中段深读区

### 9.6 终态稳定性

浏览器端展示必须遵循以下规则：

- 轮询结果在渲染前必须判定是否为“最新快照”
- `completed/failed` 终态一旦出现，不得再被旧的 `queued/running` 响应覆盖
- 展示层收到终态后，必须优先按原始终态 session 渲染，再考虑 retained result 等保留逻辑
- 即使某个次级渲染分支失败，首屏状态与主结论区也不能退回“分析中”
