# 可信投研工作台布局重构与图谱密度提升设计

## 1. 问题定义

本次变更要解决的是同一条分析链路上三个互相耦合的问题：

### 1.1 事件总览过长

当前 `event-summary` 直接复用 `summary` 结果。用户一旦粘贴很长的新闻，模型返回的摘要就会被直接放进首屏 overview card，导致：

- 第一屏被长段落吃掉
- 关键判断与证据被推到下方
- 用户看到的是“长摘要”，不是“事件概览”

这不是样式问题，而是内容层级问题。总览卡应该承载 `briefing`，不是承载完整分析文本。

### 1.2 图谱节点太少

当前抽取链路中 rich text 与 weak text 共用较强的限幅逻辑，导致：

- 强信息文本也可能只落到极少数实体与关系
- graph 页面容易看起来像“演示样例”，而不是“关系浏览器”
- 工作台内嵌图谱对判断没有形成足够的补充价值

这意味着需要把“给判断用的数据”和“给图谱用的数据”在选择层分开考虑。

### 1.3 布局不友好

当前页面仍偏向 demo 布局：

- 左栏输入过重
- 主区阅读节奏不稳定
- 右侧信息区碎片化
- 卡片持续纵向堆叠，导致页面又长又窄

需要把页面重构成真正的 result-first workbench。

## 2. 设计目标

### 2.1 目标

- 首屏在 `一屏内` 让用户读懂：发生了什么、为什么重要、凭什么相信
- 让长新闻在首屏表现为紧凑 briefing，而不是长摘要
- 让图谱在 rich article 中具备更高信息密度，但仍可控可读
- 保留弱信息文本时的稳定降级，不让节点数量增长变成噪声增长

### 2.2 非目标

- 不构建通用 knowledge graph 平台
- 不引入 graph database
- 不把工作台拆成复杂多页应用

## 3. 新的信息架构

### 3.1 首屏结构

新的页面结构调整为：

1. `顶部分析条`
2. `左侧主结果区`
3. `右侧上下文侧栏`

主结果区按顺序展示：

1. `事件总览 briefing`
2. `关键判断卡`
3. `证据摘要卡`

右侧侧栏按顺序展示：

1. `导入来源 / 运行状态 / 成本`
2. `核心主体`
3. `图谱`
4. `分析凭证`

输入区不再占据页面主要视觉，而是收缩成侧边控制台或顶部折叠入口。

### 3.1.1 首屏 wireframe（桌面端）

```text
+----------------------------------------------------------------------------------+
| 顶部分析条：来源 / 导入状态 / 运行状态 / 调用次数 / 成本 / receipt               |
+-----------------------------------------+----------------------------------------+
| 主结果区                                | 右侧栏                                 |
|                                         |                                        |
| [事件总览 briefing]                     | [导入来源 / 分析状态]                  |
|  headline + 1-2句 briefing + 指标       | [核心主体]                             |
|  CTA: 查看完整摘要                      | [辅助关系图（较大画布）]               |
|                                         | [分析凭证]                             |
| [关键判断卡 1] [关键判断卡 2]           |                                        |
| [关键判断卡 3 可选]                     |                                        |
|                                         |                                        |
| [证据摘要卡 1] [证据摘要卡 2]           |                                        |
|  CTA: 查看上下文 / 查看完整证据         |                                        |
+-----------------------------------------+----------------------------------------+
| 输入控制台（桌面端收在右侧底部或可折叠；移动端下沉到页面末尾）                   |
+----------------------------------------------------------------------------------+
```

### 3.1.2 首屏硬规则

- `事件总览` 只允许显示 `headline + briefing + metrics + 展开入口`
- `fullSummary`、完整原文证据、预置样本全文都必须进入二级层
- 首屏默认不允许出现超过 `3` 张关键判断卡和超过 `2` 张证据摘要卡
- 图谱卡必须出现在首屏可见范围内，不能继续被挤到长页面底部

### 3.2 详情交互

以下内容默认只显示摘要版本，点击后进入 detail drawer / modal：

- 关键判断卡
- 证据卡
- 预置样本卡
- 完整摘要 / 完整原文片段

目标是让首屏只显示“可扫读内容”，把完整信息移到二级层。

### 3.2.1 详情交互契约

- `关键判断卡`
  - 点击整卡或“查看详情”打开右侧 drawer（桌面）/ 全屏弹层（移动）
- `证据摘要卡`
  - 点击后进入证据详情，包含前后文和来源状态
- `预置样本卡`
  - 首屏只展示标题、来源、1 段摘要；全文与 metadata 放入详情层
- `完整摘要`
  - 不直接显示在首页，必须通过“查看完整摘要”进入详情层

关闭方式统一为：

- 右上角关闭按钮
- 背景点击关闭（modal）
- `Esc` 键关闭（桌面）

## 4. 内容模型重构

### 4.1 总览层与分析层分离

对同一条 live session，展示层需要拆成四种内容：

- `headline`
  - 用于事件标题，尽量为一句业务标题
- `briefing`
  - 1-2 句、强约束的总览摘要
- `fullSummary`
  - 完整摘要，可在详情层展开
- `evidenceContext`
  - 供证据卡和详情弹层使用的上下文片段

原则：

- `overview card` 只显示 `headline + briefing`
- `fullSummary` 不直接进入首屏
- 长新闻也只能在详情层看到完整文字

### 4.2 briefing 的生成策略

优先方案：

- 为 summary 增加更稳定的展示层派生逻辑，从完整摘要中提取 1-2 句 briefing
- 设定展示长度目标，例如 80-140 中文字
- 如果模型返回过长，则派生层强制压缩，不让 overview card 失控

这意味着“总览变短”既有模型输出策略，也有展示层兜底策略。

### 4.3 briefing 与 fullSummary 的硬规则

- `briefing` 目标为 `1-2` 句，建议控制在 `80-140` 中文字
- `fullSummary` 可以更长，但只能出现在详情层
- 当模型返回摘要过长时，展示层必须重新派生 `briefing`，而不是原样塞进 overview card

## 5. 图谱抽取与图模型重构

### 5.1 分层策略

现有链路主要把抽取结果直接映射成图。新设计改为两层：

1. `analysis entities / relations`
   - 供判断与证据使用
2. `graph candidates`
   - 供图谱使用的候选节点与候选边

第二层需要做额外的：

- ranking
- 去冗余
- 关系去噪
- 节点限幅

同时必须明确这条真实出图链路：

1. `normalizer / provider` 产生 richer entities / relations
2. `agent-session-runner` 补齐 graph candidates
3. `buildAgentGraph` 执行 selection / ranking / densify / prune
4. `live.ts / graph.ts` 只消费最终 graph view model

### 5.2 rich article 上限

对 rich article：

- 实体上限不再停留在极低数量
- 关系上限允许明显高于当前版本
- 但图谱最终可视化仍只展示优先级最高的一部分节点与边

也就是说，抽取层可以更丰富，渲染层仍需克制。

### 5.2.1 图谱密度验收目标

对 rich article，建议先将验收目标写死为：

- `核心节点`：默认 `6-10` 个
- `核心关系`：默认 `5-12` 条
- `original / derived` 展示比例：
  - 优先展示 original
  - derived 仅在 original 连通性不足时补位
- `弱数据降级`：
  - 仍可退化到清单视图或小图
  - 不为了“节点变多”而堆砌噪声

### 5.3 graph selection 规则

建议优先保留：

- 与 headline / briefing 强相关的主体
- 在多条关系中重复出现的主体
- 与结算、支付、合作、融资等事件语义紧密相关的关系

弱信号节点不直接进入主图，必要时放到详情层或降级清单里。

### 5.4 derived 边处理

derived 边保留，但必须：

- 视觉弱化
- 与 original 边分离
- 在 legend / hint 中明确解释用途

## 6. 页面布局方案

### 6.1 主阅读区

主阅读区必须更宽，卡片不再只做单列长列表，而是：

- `overview` 单独一行
- `judgment cards` 双列或 `2+1` 结构
- `evidence cards` 双列或抽屉式组织

这样能减少纵向滚动压力。

### 6.1.1 桌面端布局约束

- 主结果区最小宽度应明显大于当前版本
- 侧栏固定宽度，并在桌面端 sticky
- 输入控制台默认折叠或弱化，不再抢占首屏中心区域

### 6.2 侧栏

侧栏承担“上下文”和“工具”职责：

- 固定宽度
- 在桌面端 sticky
- 在移动端顺序下沉

其中图谱卡需要比当前更大，不再是一个狭小的次要角落。

### 6.2.1 移动端退化策略

- breakpoint 建议至少定义：
  - `>= 1280px`：主区 + 右栏 + 折叠输入控制台
  - `768px - 1279px`：主区优先，侧栏依次下沉
  - `< 768px`：详情 drawer 退化为全屏 modal，图谱卡独占一行
- 侧栏下沉顺序固定为：
  1. 导入来源 / 状态
  2. 核心主体
  3. 图谱
  4. 分析凭证
- 图谱卡的最小可读高度需要明确，避免移动端只剩一个小窗

### 6.3 图谱展示

工作台内嵌图谱与独立图谱页共用 ECharts 能力：

- 滚轮缩放
- 拖拽平移
- adjacency focus
- original / derived 样式区分

内嵌图谱用于快速浏览；
独立 graph 页用于深入查看。

## 7. 测试与验证

### 7.1 页面测试

需要新增或更新测试，锁定：

- 长新闻时 overview 只显示 briefing，而不是整段 full summary
- 详情弹层存在并能承接卡片完整信息
- 首屏不再出现过量 demo 文案
- 图谱容器尺寸和交互脚本存在

### 7.2 抽取测试

需要新增或更新测试，锁定：

- rich article 能得到更高密度的实体 / 关系
- weak article 仍保持稳定降级
- graph candidates 的去噪与限幅逻辑稳定

### 7.3 审核门

在进入实现前，先由 UI 审核员对以下内容给出意见：

- 首屏信息架构是否清晰
- 主结果区与侧栏分工是否合理
- 长摘要与长新闻是否真正被压到二级层
- 图谱是否已经有足够展示优先级

## 8. 风险

### 8.1 风险：图谱变多但更乱

应对：

- 抽取层上限放宽
- 渲染层保持更严格的优先级筛选

### 8.2 风险：总览过短导致信息不足

应对：

- overview 只承载 briefing
- 通过 detail drawer 查看 full summary

### 8.3 风险：布局重构影响当前测试

应对：

- 先补结构测试
- 再改 view model
- 最后改模板与交互
