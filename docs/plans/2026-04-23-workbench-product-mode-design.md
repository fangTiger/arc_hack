# 工作台产品模式设计记录

## 1. 结论

`/demo/live` 后续不再按“live console 演示页”定义，而统一按 `资讯分析工作台产品模式` 继续演进。

这意味着页面的核心不是展示系统能力，而是帮助用户在同一页面中完成：

- 事件理解
- 关键判断
- 证据核验
- 主体扫描
- 来源与凭证追溯

## 2. 页面骨架

桌面端统一采用四段式结构：

1. 顶部状态总控条
2. 左侧收纳式导入仓
3. 中间主结论区
4. 右侧态势侧栏

其中主区固定承载：

1. `事件总览`
2. `关键判断`
3. `证据摘录`

右侧只承载：

- `辅助关系图`
- `核心主体`
- `分析元信息入口`

## 3. 三个产品级决策

### 3.1 主结论中心

首屏采用“主结论中心”，而不是“结论 + 关系图”双核。

第一视觉必须是：

- 事件总览
- 关键判断

关系图只能增强理解，不能抢占认知中心。

### 3.2 独立证据卡区

证据摘录采用 `独立证据卡区`，紧跟在关键判断之后，不做判断卡内嵌，也不做必须点击后才出现的主方案。

目标是形成：

`看结论 -> 看依据`

的连续阅读链路。

### 3.3 产品化状态模型

页面按用户视角定义五个主状态：

- `待导入`
- `待分析`
- `分析中`
- `分析完成`
- `分析失败`

重新分析时必须保留旧结果，直到新结果生成成功；失败后也必须留在当前工作台上下文中恢复。

## 4. 视觉基调

统一定调为：

**冷静克制的专业未来感**

科技感来自秩序、结构、标签系统和轻量状态反馈，而不是霓虹、粒子、扫描线或大面积炫光。

所有正文、证据、来源、凭证区都应回到接近专业阅读器的标准，确保可核验性优先于视觉冲击。

## 5. 关键风险

需要在后续实现中持续防止三类跑偏：

1. 页面重新滑回 demo / 大屏 / 看板叙事
2. 关键判断失去证据锚点，重新变成“有观点没依据”
3. 重新分析或失败恢复时把用户打回空白页

## 6. 后续执行提示

后续实现必须优先写死三件事：

1. 页面状态与分析阶段的双层模型
2. 重新分析保留旧结果的替换规则
3. 判断、证据、来源上下文的可追溯标准
